# 用小组 PPT 理解 Git

## 传统协作的问题
在传统的艺术或学术项目中，小组合作制作幻灯片往往会陷入“命名学灾难”。
举例：我们经常会看到 `汇报PPT_A改.pptx`、`汇报PPT_B改.pptx`、`合并版.pptx`、`最终版.pptx`。这种依靠文件名来区分历史的方法，不仅容易导致版本覆写，更无法追踪“谁在何时为何做出了修改”。

## Git 怎么解决
Git 引入了平行的叙事结构：
- **分支 (Branch)**：每个人在自己的分支上制作幻灯片的特定章节（如：`feature/intro`、`feature/data-analysis`）。这就像是每个人都有了一份独立且永不丢失的草稿。
- **合并请求 (Pull Request)**：当内容准备就绪，参与者会发起 PR。这不仅是“合并代码”的请求，更是一个讨论区。

## PR Review 与合并流程
在小组协作中，Review 是确保质量的关键步骤：
1. **同行评审 (Peer Review)**：其他组员（Reviewer）会在 PR 下方留言。例如：“这里的图表颜色建议和模板保持一致”、“文案中关于调研数据的描述可以更具体一点”。
2. **迭代修改**：发起人根据建议在分支上继续提交修改，直到大家达成共识。
3. **正式合并 (Merge)**：通过 Review 后，由组长或指定人员点击 Merge，将个人贡献正式汇入主干 `main`。这种方式避免了“口头确认”导致的混乱，所有决策过程都可审计。

## 注意事项
Git 是一种基于文本差异的比对工具，不适合多人同时编辑同一个二进制的 .pptx 文件（会产生难以解决的二进制冲突）。它更适合管理项目大纲、核心文案、以及存储关键的图片素材资源。建议使用 Markdown 或 LaTeX 编写幻灯片内容，再通过 Git 协作。
